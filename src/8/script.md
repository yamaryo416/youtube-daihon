次に実装方法を二つ提示されました。
上の Subagent-Driven というのは、タスクごとに、専用の「サブエージェント」を新しく立ち上げて進めていくやり方です。

サブエージェントとは、簡単に言うと、メインの AI とは別に動く「助手」のようなものです。

1 つのタスクを、その助手に任せて、実装させて、チェックする。終わったら、次のタスクは、また別の新しい助手にバトンタッチしていく、というイメージですね。

こうすると、前のタスクの情報を引きずらないので、毎回、頭がスッキリした状態で作業できるんです。

だから、混乱やミスが起きにくく、タスクごとの品質も、1 つずつ確実に確認しながら進められるんですよね。

ですので、1がおすすめです。

superpowersのSubagent-Driven Developmentというスキルを使ってますね。

ここまで使用されたスキルのまとめです。

まず、using-superpowers。これは全体の「案内役」で、どの場面で、どのスキルを使うかを、判断してくれるスキルです。

次に、brainstorming。質問しながら、作るものの要件を一緒に整理してくれるスキルですね。

その次が、writing-plans。「計画を書く」という名前の通り、整理した要件を、具体的な実装計画に落とし込んでくれます。

そして、subagent-driven-development。こちらは「助手を使った開発」。タスクごとに専用の助手を立てて、実装とレビューを繰り返してくれるスキルです。

さらに、その実装の裏側では、test-driven-development、いわゆる TDD も自動で使われていました。先にテストを書いてから、コードを書く、という進め方ですね。

ちなみに、今回は使わなかったですが、superpowers には、git を使って作業を安全に切り分けたり、ユーザーからのレビューに適切に対応したり、バグを筋道立てて調査したりと、他にもいろいろなスキルがあります。

こんなふうに、superpowers さえ入れておけば、開発のあらゆる場面を、プロのやり方でカバーできるんです。

正直、Claude Code でアプリ開発をするなら、まず最初に入れてほしい。それくらい、主役級のプラグインです。

2 つ目は、design-taste-frontend です。

（スライドを表示）

これは「Taste Skill」というスキル集の中の 1 つです。

さっきの superpowers が、開発用の Skill をまとめた「プラグイン」だったのと同じイメージで、Taste Skill は、デザイン用の Skill を 13 個まとめた「Skill 集」。その中でも中心になるのが、今回使う design-taste-frontend です。

これは、AI が作りがちなデザインを、ちゃんと人間っぽく、自然でプロっぽい見た目に寄せてくれる Skill です。

「AI が作った画面って、なんか全部似てるな」そう思ったことありませんか？

紫のグラデーション。

中央にドンとカード。

やたら丸いボタン。

大きすぎる見出し。

なんかキラキラしてるけど、実際のサービスっぽくない画面。

ありますよね。

僕も AI で画面を作らせていて、「うわ、またこの感じか」と何度も思いました。

そこで使えるのが、design-taste-frontend です。

これは、AI にデザインの感覚を入れるための Skill です。

特に面白いのが、3 つのダイヤルで見た目の方向性を決められるところです。

このダイヤル、まず AI が「何を作るか」を読み取って、基準になる値を自動でセットしてくれます。そのうえで、こちらの好みに合わせて数字を上書きすることもできる、という仕組みです。

1 つ目が、デザイン分散。

これは、どれくらい左右対称を崩したり、レイアウトに変化をつけるか、という調整です。

2 つ目が、モーション強度。

モーションとは、画面の動きやアニメーションのことです。

どれくらい動きを入れるかを調整できます。

3 つ目が、視覚密度。

これは、1 画面にどれくらい情報を詰めるか、という考え方です。

視覚密度が低いと、余白が多くてゆったりした画面になります。

視覚密度が高いと、情報量が多く、業務ツールっぽい画面になります。

そしてこの 3 つ、スライダーやファイルをいじって動かすわけではありません。プロンプトで「視覚密度は 9 で」といったように、数字を口頭で指定するだけで調整できます。

これ、すごく実用的なんですよ。

4 つ目は、webapp-testing です。

（スライドを表示）

これは、Claude Code が実際にブラウザを操作して、アプリの動作確認をできるようにするスキルです。

これは、「anthropics/skills」というスキル集の中の 1 つ。しかも、Anthropic が公式で出しているスキルなんです。

なので、これまで紹介してきたスキルと、同じ流れで、そのまま使えます。

「コードを書いたら、ちゃんと動くか自分で確認する」

これは開発では絶対に必要です。

ただ、正直ここがめちゃくちゃ面倒なんですよ。

ログインできるか。

ボタンを押せるか。

フォームに入力できるか。

保存されるか。

ページを更新してもデータが残るか。

こういう確認を毎回手でやるの、かなり大変です。

僕も昔は、修正するたびに同じボタンを何回も押して、同じフォームに何回も入力していました。

「またこの確認か」と思いながらやっていましたね。

そこで webapp-testing です。

これを使うと、ブラウザ操作を自動化できます。

自動化とは、人間が手でやっていた作業を、コンピューターに代わりにやってもらうことです。

webapp-testing を使わせると、作ったアプリを自分で開いて、クリックして、入力して、結果を確認するところまで、Claude Code に任せられます。

■ 本編 5: まさかの OpenAI！codex:review で Claude のコードを別 AI にレビューさせる
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

最後の 5 つ目は、codex:review です。

（スライドを表示）

これは、Claude Code の中から OpenAI の Codex にコードレビューを頼めるスキルです。

そもそも Codex は、OpenAI が出している、コード作業向けの AI なんですが、面白いことに、OpenAI は、Claude Code のような「他社の AI エージェント」向けにも、プラグインを出しているんです。

そのプラグインを Claude Code に入れると、Codex のスキルが使えるようになります。

そして、その中の一つが、今回使う codex:review、というわけです。

この「わざわざ別の AI にレビューさせる」というのが、実はめちゃくちゃ理にかなっているんです。

なぜなら、同じ AI に自分のコードを自己採点させると、見落としが出ることがあるからです。

自分で書いた文章を自分で見直しても、誤字に気づきにくいのと同じです。

Claude が作ったコードを Claude だけで確認するより、別のモデルファミリーである Codex に見てもらう。

モデルファミリーとは、同じ会社や同じ設計思想で作られた AI のまとまりのことです。

別の AI に見てもらうことで、違う角度の指摘が出やすくなります。

これが、codex:review の強さです。

インストールは、

codex pluginと検索して、こちらのページに移動。

このコマンドをclaude codeのチャット欄で実行します。

これでインストールが完了です。

では、実際に使ってみます。

（ここから画面を見せながら操作）

ここまでで作った トレーニング管理アプリに対して、Codex にレビューをしてもらいます。

━━━━━━━━━━━━━━━━━
【プロンプト】

/codex:review
━━━━━━━━━━━━━━━━━

これで、Codex がコード全体を確認して、問題点を出してくれます。

ちなみに、もっと厳しく突っ込んでほしい時は、codex:adversarial-review というコマンドもあります。

これは、あえて厳しめに、「本当にこの実装で大丈夫？」と、実装判断や設計そのものにツッコミを入れてくれるレビューです。


━━━━━━━━━━━━━━━━━
【プロンプト】

/codex:adversarial-review
━━━━━━━━━━━━━━━━━

色々と問題点を見つけてくれましたね。

これ、もうほぼチーム開発なんですよ。

Claude Code が実装して、

Codex がレビューして、

その指摘を、また Claude Code が直す。

僕たち人間は、その様子を見ながら、方向性だけ決めていく。

一人で開発しているのに、まるでチームで開発しているような感覚なんです。

正直、AI に開発を任せていて一番怖いのって、「ちゃんと動いてるように見えるけど、実はどこかに穴がある」ことなんですよね。

自分ひとりだと、その穴に、なかなか気づけない。

でも、そこに別の視点を持った AI を一枚かませるだけで、気づける確率がグッと上がるんです。

もちろん、Codex のレビューも完璧ではないので、最後は人間が判断する必要があります。

それでも、自分ひとりで抱え込むより、見落としは圧倒的に減ります。

特に、「何をチェックすればいいか分からない」という初心者の方ほど、この "もう一人の厳しい目" は、心強い味方になってくれます。
